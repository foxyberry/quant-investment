# 작업 계획 문서 생성

$ARGUMENTS 에 대한 작업 계획 문서를 생성하세요.

## 작업 순서

1. **요구사항 분석**: 작업 내용 파악
2. **관련 코드 탐색**: 영향받는 파일/모듈 확인
3. **계획 수립**: 단계별 작업 계획 작성
4. **문서 생성**: 아래 템플릿으로 문서 작성
5. **Git 브랜치 생성**: 승인 후 작업용 브랜치 생성

## Git 브랜치 규칙

**중요**: 작업 시작 전 반드시 새 브랜치를 생성해야 함

### 브랜치 명명 규칙
- 이슈 기반 작업: `feature/issue{이슈번호}` (예: `feature/issue721`)
- 버그 수정: `fix/issue{이슈번호}` (예: `fix/issue123`)
- 이슈 없는 작업: `feature/{작업명}` (예: `feature/add-login`)

### 브랜치 생성 명령
```bash
git checkout -b feature/issue{이슈번호}
```

## 문서 위치

`docs/works/YYYYMMDD_작업명.md`

예: `docs/works/20260123_사용자인증기능추가.md`

## 문서 템플릿

```markdown
# 작업명

- **날짜**: YYYY-MM-DD
- **이슈**: https://github.com/foxyberry/quant-investment/issues/XXX (있으면)
- **브랜치**: `feature/issue{이슈번호}` (예: `feature/issue721`)
- **상태**: 계획중

## 목표

(이 작업으로 달성하려는 것 - 구체적으로 작성)

## 배경

(왜 이 작업이 필요한지)

## 계획

- [ ] 1단계:
- [ ] 2단계:
- [ ] 3단계:
- [ ] 테스트
- [ ] 코드 리뷰

## 변경 파일

| 파일 | 변경 내용 |
|------|----------|
| `path/to/file.kt` | 변경 설명 |

## 기술적 고려사항

(주의할 점, 의존성, 영향 범위 등)

## 결과

(완료 후 작성)
```

## 작성 가이드

### 목표
- 한 문장으로 명확하게
- 측정 가능한 결과 포함

### 계획
- 각 단계는 독립적으로 완료 가능하게
- 테스트 항목 필수 포함
- 예상 변경 파일 미리 파악

### 변경 파일
- 코드베이스 탐색 후 실제 영향받는 파일 나열
- 새로 생성할 파일도 포함

## 출력

1. 작업 계획 문서 생성
2. 파일 저장 (`docs/works/YYYYMMDD_작업명.md`)
3. 사용자에게 계획 승인 요청
4. **승인 후**: Git 브랜치 생성 (`feature/issue{이슈번호}`)

**주의**: 코드 수정 전 반드시 브랜치 생성 확인!
