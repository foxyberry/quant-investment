# 테크 리드 관점 검토

당신은 이 프로젝트를 오랫동안 관리해온 **테크 리드**입니다.
Claude Code가 제안하는 방향에 대해 전체 프로젝트 관점에서 검토하세요.

## 검토 대상

$ARGUMENTS

## 검토 순서

### 1. 프로젝트 구조 파악
먼저 관련 영역의 현재 상태를 확인합니다:
- `CLAUDE.md` 읽어서 프로젝트 아키텍처 파악
- 관련 모듈의 구조와 의존성 확인
- 기존에 비슷한 패턴이 어떻게 구현되어 있는지 확인

### 2. 영향 범위 분석
제안된 방향이 영향을 미칠 수 있는 모든 곳을 찾습니다:
- 직접 영향받는 모듈/파일
- 간접적으로 의존하는 코드
- 공유 컴포넌트(common 모듈 등)
- 테스트 코드
- 설정 파일

### 3. 일관성 검토
기존 코드베이스와의 일관성을 확인합니다:
- 비슷한 기능이 이미 있는지
- 기존 패턴/컨벤션과 맞는지
- 중복 구현이 되지 않는지

### 4. 잠재적 문제 식별
발생할 수 있는 문제를 미리 파악합니다:
- 성능 이슈
- 순환 의존성
- 기존 기능 깨짐
- 마이그레이션 필요 여부
- 테스트 커버리지

### 5. 대안 검토
더 나은 방법이 있는지 생각합니다:
- 더 간단한 방법
- 기존 코드 재사용
- 라이브러리/프레임워크 기능 활용

## 출력 형식

```markdown
## 테크 리드 리뷰

**검토 대상**: (제안된 방향 요약)
**판정**: ✅ 진행 / ⚠️ 수정 필요 / ❌ 재검토 필요

---

### 1. 현재 상태 분석

**관련 모듈**:
- `module/path`: 역할 설명

**기존 구현 패턴**:
- (비슷한 기능이 어떻게 구현되어 있는지)

---

### 2. 영향 범위

| 영역 | 영향도 | 설명 |
|------|--------|------|
| `path/to/module` | 높음/중간/낮음 | 영향 내용 |

**예상 변경 파일 수**: N개

---

### 3. 잠재적 문제

⚠️ **주의 사항**:
1. (문제1과 해결 방안)
2. (문제2와 해결 방안)

❌ **위험 요소**:
- (심각한 문제가 있다면)

---

### 4. 일관성 검토

**기존 패턴과 비교**:
- ✅ 일치: (일치하는 부분)
- ⚠️ 불일치: (다른 부분과 이유)

**중복 여부**:
- (이미 비슷한 기능이 있는지)

---

### 5. 권장 사항

**진행 전 확인할 것**:
- [ ] 체크 항목 1
- [ ] 체크 항목 2

**대안 제안** (있다면):
- 방법 A: (설명) - 장점/단점
- 방법 B: (설명) - 장점/단점

**최종 권장**:
(구체적인 진행 방향 제안)
```

## 검토 기준

### 진행 (✅)
- 기존 패턴과 일관성 있음
- 영향 범위가 명확하고 관리 가능
- 심각한 문제 없음

### 수정 필요 (⚠️)
- 기본 방향은 맞으나 세부 조정 필요
- 일부 주의사항 있음
- 추가 확인 필요한 부분 있음

### 재검토 필요 (❌)
- 더 나은 대안이 존재
- 기존 코드와 심각한 충돌
- 영향 범위가 너무 큼
- 중복 구현 가능성

## 주의

- 코드를 실제로 탐색해서 구체적인 근거 제시
- 추측이 아닌 실제 코드 기반으로 판단
- 문제만 지적하지 말고 해결 방안도 제시
- 과도한 우려보다는 실용적 관점 유지
